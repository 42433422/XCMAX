# ADR 0013：Rust 边缘网关预研门禁

- **状态**：已采纳（作为**预研门禁**，非实施承诺）
- **日期**：2026-05-08

## 背景

业界存在以 Rust 实现的高性能 API 网关或 Agent 后端（例如调研中的 OpenAgentic 一类方案）。团队在评估是否将部分高频或计算密集端点前移到 Rust。

## 决策

在未完成以下 **度量门禁** 前，**不**启动 Rust 网关 / 并行 BFF 的正式开发或拆分发布：

1. **延迟分解**：对候选端点（例如自然语言工作流图相关 HTTP API，其内部调用 `workflow_nl_graph` → LLM）采集 p50/p95，分拆为：网关、鉴权、业务逻辑、DB、外部 LLM 网络与推理。
2. **热点判定**：仅当数据表明 **Python 进程内 CPU**（序列化、图遍历、正则/JSON 巨量解析等）在总耗时中占比显著且可复现时，Rust 卸载才进入方案比选。
3. **默认结论**：`workflow_nl_graph` 等主要瓶颈通常在 **模型调用与 IO**；门禁未通过则 **文档闭环即可**，不产生 Rust 工单。

## 后果

- 架构讨论与员工（如 flask-entry-keeper、log-monitor-incident）的联动以 **ADR 门禁结果**为输入：只有通过 §2 才进入入口改造、新增进程与监控告警设计。
- 避免「为 Rust 而 Rust」的双栈运维成本。
