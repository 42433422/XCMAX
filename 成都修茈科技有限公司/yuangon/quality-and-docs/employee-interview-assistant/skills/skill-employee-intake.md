# skill-employee-intake



## 触发



用户希望为某 `pkg_id` 或某篇 runbook/技能补全信息。



## 访谈目的（融入每一次问诊）



借鉴常规员工访谈的结构化目的，使补全需求更可执行（不限顺序，按需选用）：



1. **传递信息**：向「岗位维护者」澄清本规范期望的字段含义、示例与引用位置（`employee.yaml`、`README`、`manifest.metadata`）。

2. **了解动态**：核对岗位近期职责变化、上下游协作是否变化，映射到 `depends_on`、scope/forbidden、handlers。

3. **收集问题**：显性记录阻塞项、歧义、风险与待人工裁定项，写入「缺口清单」与「依赖与风险提示」。



数字员工（Agent 类岗位）访谈时，额外强调：**能力是否与 handlers / system_prompt 一致**，避免文案与真实自动化边界脱节。



## 步骤



1. 读取（或请用户提供）目标 `employee.yaml` 与 `skills/` 列表。

2. 输出「已具备 / 缺失 / 建议填写示例」三段；缺失项用 checklist。

3. 引导填写 **能力说明** 时，优先采用下文「能力说明结构化模板」（面向 Agent/数字员工）；勿编造未提供的内部数据。

4. 若需联网检索公开最佳实践，仅作措辞参考，不冒充公司内部数据。



## 能力说明结构化模板（数字员工 / Agent）



用于 `domain`、README「职责」长段落或 `employee_config_v2.metadata.capability_summary` 类字段时，建议按块组织（可删节未适用项）：



| 维度 | 引导问题（摘要） |

|------|------------------|

| **数据处理** | 输入/输出形态（结构化文本、表格、日志、向量检索结果）；是否批处理、增量、脱敏要求。 |

| **流程自动化** | 触发方式（定时、事件、人工）；与工具/API 的编排步骤；失败重试与降级路径。 |

| **决策逻辑** | 规则 vs 模型；何种情况下拒答、转人工、引用知识库；不确定性如何声明。 |

| **协作与边界** | 依赖哪些岗位包或外部 API；与人或其它 Agent 的交接点；禁止触碰的数据域。 |



**示例草稿（可压缩为一段话）：**



> **数据处理**：接收工单 JSON，关联只读知识库切片；输出 Markdown 报告，表格字段与模板固定。  

> **流程自动化**：由编排任务触发；顺序：校验输入 → 检索 KB → 单次 LLM 调用 → 结构化校验 → 写回活动日志。  

> **决策逻辑**：事实性内容必须附引用锚点；无法查证时输出「待核实」并列出所需附加材料；高风险操作建议显式拒答并 escalate。  

> **协作与边界**：依赖 `doc-knowledge-curator` 维护的文档路径只读；禁止写 `_local_secrets/**`。



## 输出格式



- Markdown，含 `## 缺口清单`、`## 建议草稿`、`## 依赖与风险提示`。

- 「建议草稿」中若涉及能力描述，应体现上表至少两个维度；Agent 类岗位建议四维齐全或注明「不适用」及原因。
